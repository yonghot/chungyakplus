# 청약메이트(ChungYakMate) 기본지침서

> 시행사 도메인 지식 기반 청약 자격 진단 및 당첨 확률 분석 서비스
> 기술 스택: Next.js (App Router), Vercel, Supabase, shadcn/ui, Tailwind CSS

---

## 1. 언어 원칙

사고 과정을 포함한 모든 작업에서 한글을 사용한다. 코드 주석, 커밋 메시지, 문서, 대화 응답 모두 한국어로 작성한다. 변수명/함수명/클래스명 등 코드 식별자는 영어(camelCase/PascalCase)를 사용한다. 에러 메시지 중 사용자 노출용은 한국어, 내부 로그는 영어를 허용한다.

### 실행 규칙

- 도메인 용어는 한영 매핑 테이블(`types/glossary.ts`)에 정의하고, 코드에서는 영어 식별자를, UI/메시지에서는 한국어를 쓴다 (예: `specialSupply` → "특별공급")
- PR 제목, 브랜치 설명, 이슈 본문 모두 한국어로 작성한다
- JSDoc/TSDoc 주석의 `@param`, `@returns` 설명은 한국어로 쓴다
- 외부 라이브러리 에러를 사용자에게 전달할 때는 반드시 한국어로 번역된 메시지를 사용한다
- Supabase RLS 정책명, DB 함수명은 영어 snake_case를 사용하되, 해당 마이그레이션 파일 상단에 한국어 주석으로 목적을 명시한다

## 2. 코드 작성 원칙

구글 TypeScript 스타일 가이드를 기본으로 따른다. ESLint + Prettier 설정을 프로젝트 초기에 구성하고 일관되게 유지한다. 함수는 단일 책임 원칙을 지킨다. 함수 50줄, 파일 300줄을 넘으면 분리를 검토한다.

**린트 규칙**: `no-any`, `no-console`, `prefer-const`, `no-unused-vars`

**네이밍 규칙**:
| 대상 | 컨벤션 | 예시 |
|------|--------|------|
| 컴포넌트 | PascalCase | `EligibilityCard` |
| 유틸/훅 | camelCase | `useSubscription`, `formatDate` |
| 상수 | UPPER_SNAKE_CASE | `MAX_HOUSEHOLD_MEMBERS` |
| 타입/인터페이스 | PascalCase (접미사 생략) | `Subscription`, `HouseholdMember` |
| API 라우트 | kebab-case | `app/api/eligibility-check/` |

### 실행 규칙

- `any` 타입 사용을 전면 금지한다. 타입을 모를 때는 `unknown`을 쓰고 타입 가드로 좁힌다
- 컴포넌트 props는 반드시 인터페이스로 정의하고, 컴포넌트 파일 상단 또는 `types/` 디렉토리에 분리한다
- `console.log`는 개발 중에도 사용하지 않는다. 디버깅은 `lib/logger.ts` 래퍼를 통해 환경별로 제어한다
- Supabase 쿼리 결과는 항상 에러 핸들링한다: `const { data, error } = ...` 후 `error` 체크를 빠뜨리지 않는다
- 서버 컴포넌트와 클라이언트 컴포넌트를 명확히 구분한다. `'use client'` 선언은 실제 클라이언트 기능(hooks, 이벤트 핸들러)이 필요한 최소 단위에만 붙인다

## 3. 아키텍처 원칙 — 모듈화와 독립성

모든 기능을 적절한 단위로 나누어 모듈화하고, 각 모듈에 명확한 이름을 부여한다. 특정 기능 수정 시 다른 모듈에 영향이 없도록 독립성을 확보한다. RESTful API 설계 원칙을 절대 준수한다(리소스 중심 URL, HTTP 메서드 의미 준수, 적절한 상태 코드). 내부 API를 활용하여 모듈 간 통신은 API를 통한다.

**레이어 간 의존 방향**: 컴포넌트 → 서비스 → 리포지토리 (역방향 금지)

**디렉토리 구조**:
```
app/              → 라우팅, 페이지, API 라우트
  api/[리소스]/   → API 엔드포인트
components/       → UI 컴포넌트 (공용 / 도메인별)
lib/
  services/       → 비즈니스 로직
  repositories/   → 데이터 접근 계층 (Supabase 쿼리)
  utils/          → 순수 유틸리티 함수
types/            → 타입 정의
hooks/            → 커스텀 훅
constants/        → 상수 및 설정값
```

### 실행 규칙

- 서비스 레이어(`lib/services/`)는 Supabase 클라이언트를 직접 호출하지 않는다. 반드시 리포지토리(`lib/repositories/`)를 통해 데이터에 접근한다
- API 라우트 핸들러(`app/api/`)는 요청 파싱과 응답 포맷팅만 담당한다. 비즈니스 로직은 서비스 레이어에 위임한다
- 컴포넌트에서 Supabase를 직접 호출하지 않는다. 훅 또는 서버 액션을 통해 서비스 레이어를 호출한다
- 순환 의존이 발생하면 즉시 수정한다. 공유 타입은 `types/`에, 공유 유틸은 `lib/utils/`에 배치한다
- 청약 도메인 로직(자격 판정, 가점 계산 등)은 `lib/services/eligibility/` 아래에 기능 단위로 분리하고, 각 모듈은 독립적으로 테스트 가능해야 한다

## 4. 최소 구현 원칙 — 보수적 개발

요청한 사항에 대해서만 최소한으로 보수적으로 구현한다. 임의로 관련 기능을 추가 구현하지 않는다. 요청하지 않은 UI 개선, 추가 API, 부가 기능을 만들지 않는다. 변경 범위를 최소화하여 요청된 파일만 수정한다. 리팩터링은 먼저 제안하고 승인 후에 진행한다.

### 실행 규칙

- 새 파일을 생성하기 전에 기존 파일에 추가할 수 있는지 먼저 검토한다
- 기존 코드를 수정할 때, 변경과 무관한 코드 스타일/포맷 변경을 하지 않는다
- "나중에 필요할 것 같은" 파라미터, 옵션, 확장 포인트를 미리 만들지 않는다
- 새로운 의존성(npm 패키지) 추가가 필요하면 반드시 사전에 알리고 승인을 받는다
- 하나의 작업 요청에 대해 하나의 논리적 변경만 수행한다. 부수적 개선을 발견하면 별도로 제안한다

## 5. 트러블슈팅 원칙 — 근본 원인 해결

근본 원인을 파악하여 재발을 방지한다. "일단 돌아가게 하기"를 금지한다.

**디버깅 프로세스**:
1. 에러 메시지를 정확히 읽는다
2. 재현 조건을 파악한다
3. 근본 원인을 특정한다
4. 최소 범위로 수정한다
5. 재발 방지 지침을 이 문서 9절에 추가한다

**금지 패턴**: try-catch로 에러 삼키기, 조건문 우회, `any` 타입 캐스팅, 기능 비활성화

### 실행 규칙

- 에러를 catch할 때는 반드시 로깅하거나 상위로 전파한다. 빈 catch 블록을 허용하지 않는다
- 타입 에러를 `as any`나 `@ts-ignore`로 해결하지 않는다. 타입 정의를 수정한다
- 수정 전후의 동작을 확인할 수 있는 방법(테스트, 수동 검증 절차)을 반드시 제시한다
- 동일 버그가 다른 곳에서도 발생할 수 있는지 검색하고, 있으면 함께 수정한다
- Supabase 관련 에러는 RLS 정책, 스키마 불일치, 인증 상태를 순서대로 점검한다

## 6. 토큰 효율 원칙 — 낭비 최소화

별도 .md 가이드 파일로 불필요하게 파일 개수를 늘리지 않는다. 문서는 핵심만 간결하게 작성하고, 코드 예시가 필요하면 파일 경로를 안내한다. 주석은 "무엇을"이 아닌 "왜"를 쓴다. 명확한 네이밍으로 코드 자체가 설명이 되게 한다.

### 실행 규칙

- 이 CLAUDE.md 하나로 프로젝트 지침을 관리한다. 별도의 CONTRIBUTING.md, STYLE_GUIDE.md 등을 만들지 않는다
- 응답에서 코드 블록을 쓸 때 변경되지 않는 부분은 생략하고, 변경 포인트만 보여준다
- 타입이나 함수 시그니처가 명확하면 JSDoc을 생략한다. 비즈니스 규칙이 복잡한 경우에만 주석을 단다
- 파일 상단의 보일러플레이트 주석(작성자, 날짜 등)을 쓰지 않는다. Git 히스토리가 그 역할을 한다
- 동일한 설명을 여러 파일에 중복 작성하지 않는다. 한 곳에 정의하고 참조한다

## 7. 과도한 일반화 금지

특정 케이스용 로직을 다른 필드에 임의 적용하지 않는다. YAGNI 원칙을 따르며 선제적 추상화를 금지한다. 제네릭/유틸리티는 3곳 이상에서 사용할 때에만 일반화한다. 설정/옵션 객체를 과도하게 유연하게 만들지 않는다.

### 실행 규칙

- 청약 유형(일반공급, 특별공급 등)별 자격 판정 로직은 각각 독립된 함수로 구현한다. 공통 로직이 명확해질 때까지 추상화하지 않는다
- 컴포넌트 props에 `variant`, `mode` 같은 분기 옵션을 3개 이상 두지 않는다. 별도 컴포넌트로 분리한다
- 유틸리티 함수를 만들기 전에 실제로 2곳 이상에서 동일 패턴이 반복되는지 확인한다
- 미래의 요구사항을 예측해 인터페이스에 옵셔널 필드를 미리 추가하지 않는다
- 커스텀 훅은 실제 재사용이 필요할 때만 추출한다. 컴포넌트 내부 로직이 복잡하다고 무조건 훅으로 분리하지 않는다

## 8. 버전 관리 원칙

커밋 메시지는 `[타입] 한국어 설명` 형식을 따른다. 하나의 커밋 = 하나의 논리적 변경이다. 빌드가 깨지는 상태로 커밋하지 않는다.

**타입**: `feat`, `fix`, `refactor`, `docs`, `style`, `test`, `chore`

### 실행 규칙

- 커밋 전 `npm run build`가 성공하는지 확인한다
- 마이그레이션 파일 변경은 별도 커밋으로 분리한다 (예: `[chore] 청약 자격 테이블 스키마 추가`)
- 여러 파일에 걸친 리팩터링이라도 하나의 논리적 변경이면 하나의 커밋으로 묶는다
- 환경변수(`.env`) 관련 변경은 `.env.example`을 함께 업데이트한다
- Supabase 마이그레이션 파일은 순서가 중요하므로 타임스탬프 기반 파일명 규칙을 지킨다

## 9. 프로젝트 진행 중 추가되는 원칙

> 트러블슈팅 교훈 및 재발 방지 지침이 여기에 누적된다.

### 9.1 오픈 리다이렉트 방지
- **교훈**: `searchParams.get('redirect')`를 그대로 `router.push()`에 전달하면 외부 URL로의 리다이렉트 공격 가능
- **재발 방지**: 리다이렉트 경로는 반드시 `/`로 시작하고 `//`로 시작하지 않는 상대 경로만 허용. 검증 실패 시 기본 경로(`/complexes`) 사용

### 9.2 시크릿 비교 시 타이밍 공격 방지
- **교훈**: `===` 연산자를 사용한 문자열 비교는 타이밍 사이드 채널 공격에 취약
- **재발 방지**: API 시크릿, 토큰 비교에는 반드시 `crypto.timingSafeEqual()` 사용. 길이가 다른 경우에도 일정 시간이 소요되도록 패딩 처리

### 9.3 서비스 간 상수 정합성 유지
- **교훈**: `profile-service.ts`와 `eligibility-service.ts`가 프로필 완성도 임계값을 독립적으로 정의하여 60% vs 80%로 불일치 발생
- **재발 방지**: 여러 서비스에서 공유하는 임계값/상수는 `constants/` 디렉토리에 단일 소스로 관리하거나, 관련 모듈 간 참조 관계를 명시

### 9.4 환경변수 Non-null 단언 금지
- **교훈**: `process.env.NEXT_PUBLIC_*!` 형태의 non-null 단언은 런타임 에러를 숨김
- **재발 방지**: 환경변수 접근 시 명시적 null 체크 후 의미 있는 에러 메시지와 함께 `throw`. `.env.example`에 모든 필수 변수를 문서화

### 9.5 테스트 기대값은 구현체 기준으로 작성
- **교훈**: `formatArea(60)`이 `'60m²'`를 반환할 것으로 가정했으나, 구현체는 `.toFixed(2)`를 사용하여 `'60.00m²'` 반환
- **재발 방지**: 테스트 작성 전 반드시 구현체 코드를 확인하고, 구현체의 실제 동작에 맞는 기대값을 설정. "있어야 할 동작"이 아닌 "실제 동작"을 테스트
