# 청약메이트 PRD 분석서

> **작성일**: 2026-03-04
> **분석 대상**: PRD v1.0
> **상태**: 개발 착수 준비 완료

---

## 1. 제품 핵심 가치 정의

**한 문장 정의**: 시행사 도메인 지식 기반의 법령 규칙 엔진을 통해 청약자가 7개 공급유형별 자격 적합성을 3초 이내에 정확히 판정받고, 맞춤 단지 추천까지 원스톱으로 제공받는 청약 의사결정 지원 서비스.

**핵심 차별화 요소**:
- 단순 정보 제공이 아닌 **법령 기반 규칙 엔진**을 통한 정확한 적격/부적격 판정
- 시행사 감수 피드백과 실제 판정 결과 축적을 통한 **자기 보정 아키텍처**
- 7개 공급유형 동시 판정으로 **최적 청약 전략** 수립 지원

---

## 2. 제품 비전 및 성공 지표 (KPI)

### 비전
연간 2만 건의 부적격 당첨 취소 문제를 해결하여, 모든 청약자가 자신의 자격을 정확히 이해하고 최적의 청약 전략을 수립할 수 있는 국민 청약 플랫폼.

### MVP KPI

| 지표 | 목표치 | 측정 방식 | 비고 |
|------|--------|-----------|------|
| MAU | 10,000명 | Supabase Auth 기반 활성 사용자 | MVP 6개월 목표 추정 |
| 판정 정확도 | 95% (MVP) → 99% (정식) | 실제 당첨/부적격 결과 대비 판정 일치율 | 3차 축적 단계에서 보정 |
| 판정 소요 시간 | < 3초 (p95) | 서버 응답 시간 | Edge Function 활용 시 개선 여지 |
| 프로필 완성률 | 70% | 필수 필드 입력 완료 비율 | 온보딩 위자드 UX가 핵심 |
| 월간 판정 횟수 | 50,000건 | 판정 API 호출 수 | MAU 대비 인당 5회 판정 |

### 보조 지표 (PRD에 명시되지 않았으나 추적 권장)

| 지표 | 권장 목표 | 근거 |
|------|-----------|------|
| 온보딩 완료율 | > 60% | 4단계 위자드 이탈률 관리 |
| 재방문율 (7일) | > 40% | 추천/알림 기능 효과 측정 |
| 판정→북마크 전환율 | > 20% | 판정 결과의 실질적 활용도 |

[미확인: MAU 10,000명의 달성 기간(타임라인)이 명시되지 않음. 합리적 가정으로 MVP 출시 후 6개월 기준으로 산정]

## 5. P0 기능 상세 분석

### F-001: 사용자 인증

**사용자 스토리**:
"김민준(청약 초보자)으로서, 이메일이나 카카오로 간편하게 가입하고 싶다, 왜냐하면 복잡한 가입 절차는 서비스 이용을 포기하게 만들기 때문이다."

**인수 조건 (AC)**:
1. 이메일/비밀번호 또는 카카오 소셜 로그인으로 가입할 수 있다
2. 로그인 후 JWT 기반 세션이 유지되며, 보호 라우트에 접근할 수 있다
3. 미인증 사용자가 보호 라우트 접근 시 `/login?redirect={원래경로}`으로 리다이렉트된다
4. 이미 인증된 사용자가 `/login`, `/signup` 접근 시 `/complexes`로 리다이렉트된다
5. 로그아웃 시 세션이 즉시 무효화되고 공개 페이지로 리다이렉트된다

**구현 상세**:
- Supabase Auth 활용 (이메일 + 카카오 OAuth)
- Next.js `middleware.ts`에서 세션 검증 및 리다이렉트 처리
- JWT 토큰은 httpOnly 쿠키로 관리 (Supabase SSR 헬퍼)
- Server Actions에서 `createServerClient`로 서버사이드 인증

**기술적 고려사항**:
- [미확인: 카카오 이외 추가 소셜 로그인(구글, 네이버) 지원 여부. 합리적 가정 — MVP에서는 카카오만 지원하되 확장 가능한 구조로 설계]
- [미확인: 비밀번호 찾기/재설정 플로우 상세. 합리적 가정 — Supabase Auth 기본 이메일 기반 비밀번호 재설정 사용]
- [미확인: 이메일 인증(verify) 필수 여부. 합리적 가정 — MVP에서는 이메일 인증 없이 즉시 사용 가능, 정식 출시 시 적용]

### F-002: 사용자 프로필 시스템

**사용자 스토리**:
"이수진(재도전자)으로서, 나의 세대 정보를 한 번 정확하게 입력하면 여러 단지에 반복 적용하고 싶다, 왜냐하면 매번 같은 정보를 입력하는 것은 시간 낭비이기 때문이다."

**인수 조건 (AC)**:
1. 온보딩 위자드(4단계: 기본정보 → 세대정보 → 자산정보 → 청약이력)로 프로필을 생성할 수 있다
2. 프로필 완성도가 퍼센트로 표시되며, 미입력 필드에 대한 안내가 제공된다
3. 프로필 수정 시 이전 판정 결과에 영향을 주는 필드 변경 시 재판정 안내가 표시된다
4. 입력값은 zod 스키마로 실시간 검증되며, 잘못된 입력 시 구체적 오류 메시지가 표시된다
5. 온보딩 중간 이탈 시 입력 데이터가 보존되어 이어서 진행할 수 있다

**온보딩 위자드 단계별 필드**:

| 단계 | 필드 | 타입 | 필수 | 검증 규칙 |
|------|------|------|------|-----------|
| **Step 1: 기본정보** | 이름 | text | O | 2~20자 한글/영문 |
| | 생년월일 | date | O | 만 19세 이상 |
| | 연락처 | tel | X | 010-XXXX-XXXX |
| **Step 2: 세대정보** | 세대주 여부 | boolean | O | - |
| | 혼인 상태 | select | O | 미혼/기혼/이혼/사별 |
| | 혼인일 | date | 조건부 | 기혼 시 필수 |
| | 부양가족 수 | number | O | 0~10 |
| | 무주택 시작일 | date | X | 과거 날짜만 허용 |
| **Step 3: 자산정보** | 총자산 | currency | X | 0 이상 정수 |
| | 월소득 | currency | X | 0 이상 정수 |
| | 자동차가액 | currency | X | 0 이상 정수 |
| **Step 4: 청약이력** | 통장 유형 | select | X | 청약저축/청약예금/주택청약 |
| | 통장 가입일 | date | X | 과거 날짜만 허용 |
| | 납입 횟수 | number | X | 0 이상 정수 |
| | 당첨 이력 여부 | boolean | O | - |
| | 당첨일 | date | 조건부 | 당첨 이력 있을 시 필수 |

**프로필 완성도 계산**:
- 필수 필드 전체 입력: 기본 60%
- 선택 필드 입력 비율에 따라 나머지 40% 배분
- 판정에 필수적인 필드 미입력 시 해당 공급유형 판정 불가 안내

**기술적 고려사항**:
- `react-hook-form` + `zod`로 다단계 폼 상태 관리
- `onboarding-store.ts`에서 임시 상태 관리 (zustand 또는 sessionStorage)
- [미확인: 부양가족의 상세 정보(각 가족 구성원의 나이, 관계 등) 입력 필요 여부. 합리적 가정 — MVP에서는 부양가족 수만 입력, 정식 출시 시 상세 구성원 정보 추가]
- [미확인: 자산/소득 정보의 검증 수단(공적 증빙 연동 등). 합리적 가정 — MVP에서는 자기 신고 기반, 향후 마이데이터 연동 고려]

### F-003: 청약 자격 판정 엔진 (핵심 도메인)

**사용자 스토리**:
"김민준(청약 초보자)으로서, 특정 단지의 7개 공급유형별 자격 여부를 한눈에 정확하게 알고 싶다, 왜냐하면 부적격 당첨 취소를 사전에 방지하고 싶기 때문이다."

**인수 조건 (AC)**:
1. 일반공급, 신혼부부, 생애최초, 다자녀, 노부모부양, 기관추천, 이전기관 7개 유형에 대해 적격/부적격/조건부 판정을 수행한다
2. 판정 결과에 근거 법령 조항과 부적격 사유가 명시된다
3. 판정 소요 시간이 3초 이내이다 (p95)
4. 동일 입력에 대해 동일 결과를 보장한다 (멱등성)
5. 프로필 미완성 시 판정 불가 사유를 구체적으로 안내하고 프로필 편집으로 유도한다
6. 각 공급유형별 판정 규칙이 독립적으로 실행되며, 하나의 유형 판정 실패가 다른 유형에 영향을 주지 않는다

**판정 엔진 아키텍처 (3단계)**:

```
1차 법령 기반
├── 주택공급에 관한 규칙
├── 주택법
├── 소득세법
└── 각 공급유형별 시행령/시행규칙

2차 시행사 감수
├── 시행사 클라이언트 구두 피드백
├── 실무 해석 차이 반영
└── 예외 케이스 규칙 추가

3차 운영 축적
├── 실제 판정 결과 수집
├── 정확도 모니터링
└── 규칙 자동 보정 (향후)
```

**7개 공급유형별 주요 판정 조건**:

| 공급유형 | 주요 자격 조건 | 근거 법령 |
|----------|---------------|-----------|
| **일반공급** | 무주택 세대구성원, 청약통장 가입 + 지역별 납입 기간 충족 | 주택공급규칙 제25조 |
| **신혼부부** | 혼인 기간 7년 이내, 무주택, 소득 기준 충족 | 주택공급규칙 제35조 |
| **생애최초** | 생애 최초 주택 구입, 무주택, 소득 기준 충족, 5년 이상 소득세 납부 | 주택공급규칙 제36조 |
| **다자녀** | 미성년 자녀 3명 이상, 무주택 | 주택공급규칙 제37조 |
| **노부모부양** | 만 65세 이상 직계존속 3년 이상 부양, 무주택 세대주 | 주택공급규칙 제38조 |
| **기관추천** | 해당 기관 추천 대상 (장애인, 국가유공자 등) | 주택공급규칙 제39조 |
| **이전기관** | 수도권 이전 공공기관 종사자 | 주택공급규칙 제40조 |

**판정 결과 구조**:

```typescript
interface EligibilityResult {
  supply_type: SupplyType;
  result: 'eligible' | 'ineligible' | 'conditional';
  score?: number; // 가점 (일반공급 시)
  reasons: {
    rule_key: string;
    rule_name: string;
    passed: boolean;
    law_reference: string;
    detail: string; // 부적격 사유 또는 충족 근거
  }[];
  evaluated_at: string;
}
```

**기술적 고려사항**:
- `lib/eligibility/rules/` 하위에 공급유형별 독립 모듈로 구현
- 각 규칙 모듈은 동일한 인터페이스를 구현하여 엔진에서 일괄 호출
- `eligibility_rules` 테이블에 JSON 기반 규칙을 저장하여 코드 배포 없이 규칙 조정 가능
- [미확인: 규칙 변경 이력 관리(버전닝) 방식. 합리적 가정 — `eligibility_rules` 테이블의 `is_active` 플래그와 `updated_at`으로 기본 이력 관리, 별도 히스토리 테이블은 P1에서 구현]
- [미확인: '조건부(conditional)' 판정의 구체적 기준과 사용자에게 표시할 안내 메시지. 합리적 가정 — 일부 조건 충족/일부 미확인 시 적용하며, 미충족 조건 목록과 함께 "추가 정보 입력 시 판정 가능" 안내]
- [미확인: 판정 규칙 우선순위(priority) 필드의 구체적 활용 방식. 합리적 가정 — 낮은 priority 값부터 먼저 평가하며, 핵심 자격 요건(무주택 등)이 먼저 평가되어 조기 부적격 반환(early return) 가능]

### F-004: 가점 자동 계산기

**사용자 스토리**:
"이수진(재도전자)으로서, 나의 청약 가점을 정확하게 자동 계산하여 경쟁력을 파악하고 싶다, 왜냐하면 수동 계산 실수로 3번이나 낙첨했기 때문이다."

**인수 조건 (AC)**:
1. 무주택기간(32점), 부양가족수(35점), 청약통장 가입기간(17점) 항목별 점수가 자동 계산된다
2. 각 항목별 점수 산출 근거가 표시된다 (예: "무주택기간 8년 3개월 → 18점")
3. 프로필 변경 시 가점이 실시간 재계산된다
4. 총점 84점 만점 기준으로 백분위 또는 경쟁력 수준이 표시된다

**가점 체계 상세 (84점 만점)**:

| 항목 | 만점 | 산출 기준 | 프로필 소스 필드 |
|------|------|-----------|-----------------|
| 무주택 기간 | 32점 | 만 30세 이후 무주택 기간 (1년 미만 2점 ~ 15년 이상 32점) | `birth_date`, `homeless_start_date` |
| 부양가족 수 | 35점 | 0명 5점 ~ 6명 이상 35점 | `dependents_count` |
| 청약통장 가입기간 | 17점 | 6개월 미만 1점 ~ 15년 이상 17점 | `subscription_start_date` |

**기술적 고려사항**:
- `lib/eligibility/scoring.ts`에 순수 함수로 구현
- 가점 산출표를 상수로 관리하여 변경 용이
- 프로필 변경 감지 시 TanStack Query invalidation으로 재계산 트리거
- [미확인: 만 30세 미만 무주택기간 산정 방식(미혼 세대원의 경우 30세 이전 기간은 무주택기간 미산입이 법령 기준이나 예외 존재). 합리적 가정 — 법령 기준대로 만 30세부터 산정하되, 예외 케이스(배우자 무주택기간 합산 등)는 2차 감수에서 보정]

### F-005: 모집공고 데이터 수집 및 표시

**사용자 스토리**:
"김민준(청약 초보자)으로서, 현재 접수 중인 모집공고를 한 곳에서 쉽게 확인하고 싶다, 왜냐하면 여러 사이트를 돌아다니며 공고를 찾는 것이 번거롭기 때문이다."

**인수 조건 (AC)**:

4. 단지 목록은 페이지네이션 또는 무한 스크롤로 표시된다
5. 단지를 북마크하여 관심 단지로 저장할 수 있다

**MVP 데이터 전략**:

**기술적 고려사항**:
- ISR (Incremental Static Regeneration)으로 단지 상세 페이지 캐싱 (`revalidate: 3600`)
- TanStack Query로 단지 목록 클라이언트 캐싱 및 필터링
- [미확인: MVP 시드 데이터의 구체적 범위(어느 지역, 몇 건). 합리적 가정 — 서울/경기 수도권 위주 15~20건의 실제 모집공고 기반 데이터]

## 6. 사용자 플로우

### 해피 패스: 첫 판정까지

```
[Step 1] 랜딩 페이지 (/)
    │  사용자가 "청약 자격 확인하기" 버튼 클릭
    ▼
[Step 2] 회원가입 (/signup)
    │  이메일 또는 카카오로 가입
    │  ├── 이메일: 이메일 + 비밀번호 입력 → 가입 완료
    │  └── 카카오: OAuth 플로우 → 동의 → 가입 완료
    ▼
[Step 3] 온보딩 위자드 (/onboarding)
    │  Step 1: 기본정보 (이름, 생년월일, 연락처)
    │  Step 2: 세대정보 (세대주여부, 혼인상태, 부양가족수, 무주택기간)
    │  Step 3: 자산정보 (총자산, 월소득, 자동차가액)
    │  Step 4: 청약이력 (통장유형, 가입일, 납입횟수, 당첨이력)
    │  → 프로필 저장
    ▼
[Step 4] 프로필 확인 (/profile)
    │  완성도 100% 확인
    │  가점 자동 계산 결과 표시 (예: 52/84점)
    ▼
[Step 5] 단지 목록 탐색 (/complexes)
    │  지역/상태 필터로 관심 단지 탐색
    ▼
[Step 6] 단지 상세 확인 (/complexes/:id)
    │  공급유형별 세대수, 면적, 가격 확인
    │  "자격 판정하기" 버튼 클릭
    ▼
[Step 7] 자격 판정 결과 (/complexes/:id/eligibility)
    │  7개 공급유형별 적격/부적격/조건부 결과 표시
    │  적격 유형의 가점 및 근거 법령 확인
    │  → 북마크 또는 추천 단지 확인으로 이동
    ▼
[Step 8] 맞춤 추천 (/recommend)
    │  적격 판정된 단지 목록 확인
    │  알림 설정
```

### 에러 패스

| 상황 | 감지 시점 | 처리 방법 | 사용자 경험 |
|------|-----------|-----------|-------------|
| 프로필 미완성 상태에서 판정 시도 | 판정 API 호출 시 | `PROFILE_INCOMPLETE` 에러 반환 | 누락 필드 목록 + "프로필 완성하기" 버튼 |
| 판정 엔진 타임아웃 (>3초) | 서버 응답 타임아웃 | `EVALUATION_TIMEOUT` 에러 반환 | "잠시 후 다시 시도해주세요" 토스트 + 재시도 버튼 |
| 잘못된 프로필 데이터 | 폼 제출 시 | zod 검증 실패 | 인라인 오류 메시지 (필드 하단) |
| 미인증 상태 보호 라우트 접근 | middleware.ts | 세션 부재 감지 | `/login?redirect={원래경로}` 리다이렉트 |
| 네트워크 오류 | API 호출 실패 | TanStack Query retry(3회) | 오프라인 안내 배너 + 재시도 |
| 단지 데이터 없음 | 단지 목록 조회 | 빈 결과 반환 | "아직 등록된 단지가 없습니다" 빈 상태 UI |
| 중복 판정 요청 | 동일 (프로필, 단지, 유형) 조합 | UNIQUE 제약 조건 → upsert | 기존 결과 갱신 (최신 판정으로 대체) |

## 12. 아키텍트에게 전달할 핵심 의사결정 사항

### 1. 판정 엔진 아키텍처 설계

**결정 필요**: 규칙 엔진의 실행 위치와 방식
- **Option A**: 서버사이드 순수 TypeScript 함수 (Server Actions/API Route)
- **Option B**: Supabase Edge Function (Deno)
- **Option C**: PostgreSQL 함수 (plpgsql)

**고려사항**:
- 7개 공급유형 × N개 규칙 = 판정 시 다수 규칙 평가 필요
- p95 < 3초 성능 요구사항 충족 필요
- `eligibility_rules` 테이블의 JSON 규칙과 코드 기반 규칙의 혼합 전략
- 규칙 변경 시 코드 배포 없이 반영 가능해야 함 (2차 감수 반영)

**권장**: Option A (TypeScript 함수) + `eligibility_rules` 테이블 참조 하이브리드. 규칙 로직은 코드로, 규칙 파라미터(임계값 등)는 DB에서 로드.

---

### 2. 규칙 데이터 모델 vs 코드 로직 경계

**결정 필요**: `eligibility_rules.condition` JSONB 필드의 활용 범위
- **Option A**: 모든 규칙을 JSONB 조건으로 표현 (No-Code 규칙 엔진)
- **Option B**: 핵심 로직은 코드, 파라미터/임계값만 JSONB
- **Option C**: JSONB는 메타데이터 전용, 모든 로직은 코드

**고려사항**:
- 청약 판정 규칙은 단순 비교가 아닌 복합 조건 (기간 계산, 소득 구간 등)
- 시행사 감수 피드백으로 규칙 변경 빈도 예상
- 개발자가 아닌 운영자의 규칙 수정 필요성
- MVP 95% 정확도 달성을 위한 규칙 정밀도 요구

**권장**: Option B. 복잡한 판정 로직(기간 계산, 가족 관계 판단 등)은 TypeScript 모듈로, 소득 기준액/기간 임계값 등 변경 가능한 파라미터는 DB JSONB에 저장.

---

### 3. 가점 계산 모듈 위치

**결정 필요**: 가점 계산의 실행 위치
- **Option A**: 클라이언트 사이드 (프로필 입력 시 즉시 계산)
- **Option B**: 서버 사이드 (판정 API 호출 시 함께 계산)
- **Option C**: 양쪽 모두 (클라이언트 미리보기 + 서버 확정)

**고려사항**:
- "프로필 변경 시 가점이 실시간 재계산된다" (AC-3)
- 가점 계산은 순수 함수 (외부 의존 없음)
- 서버 결과가 권위적(authoritative) 값이어야 함

**권장**: Option C. `lib/eligibility/scoring.ts`를 서버/클라이언트 공유 모듈로 작성. 클라이언트에서 미리보기 표시, 판정 API 호출 시 서버에서 최종 확정.

### 4. 인증 미들웨어 전략

**결정 필요**: 라우트 보호 및 리다이렉트 로직 구현 방식
- **Option A**: Next.js middleware.ts에서 모든 라우트 보호
- **Option B**: 레이아웃 컴포넌트에서 서버사이드 검증
- **Option C**: 혼합 (middleware에서 기본 보호 + 레이아웃에서 세분화)

**고려사항**:
- 프로필 미생성 시 `/onboarding`으로 리다이렉트 (세션 + 프로필 존재 여부 확인 필요)
- 인증 사용자의 `/login` 접근 시 리다이렉트
- Edge Runtime에서 Supabase 세션 검증 가능 여부

**권장**: Option C. middleware.ts에서 JWT 유효성 검증 + 기본 리다이렉트, (main) layout.tsx에서 프로필 존재 여부 확인 + 온보딩 리다이렉트.

### 5. 모집공고 데이터 수집 전략 (MVP → 정식)

- 공공데이터포털 API 또는 청약홈 크롤링 방식 선택

**권장**: MVP 시드 데이터를 실제 공공데이터 API 응답 형식에 맞춰 작성하여 전환 비용 최소화. `raw_data` 필드에 원본 응답 보존.

### 6. 알림 시스템 확장 설계

**결정 필요**: 인앱 알림(MVP)에서 외부 채널(이메일/SMS/푸시)로의 확장 경로

- 배치 알림 생성의 트리거 메커니즘

**권장**: MVP에서 `notifications` 테이블 + polling 방식. Realtime은 성능 검증 후 적용. 알림 생성은 Supabase Database Function + trigger 또는 별도 서비스로.
